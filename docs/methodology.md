# Methodology

This document describes how the knowledge graph was constructed and how the tutor uses it. It's written for a reader familiar with general ML/NLP but not the specific design choices in this project.

## Construction pipeline

### Phase 1: Exploring co-occurrence (rejected)

The initial approach crawled Wikipedia, treating pages as nodes and internal links as edges. This produced a graph of ~1,000 nodes and ~59,000 edges within a few hours.

**Why it didn't work:** the resulting graph was a co-mention network, not a knowledge graph. The edge "Algebra → Mathematics" is technically present in Wikipedia but conveys nothing pedagogically: it doesn't tell you whether algebra requires mathematics, generalizes from it, or merely shares a category. Without typed edges, the graph couldn't power any meaningful downstream task.

This experiment is preserved in `notebooks/01_wikipedia_crawler.ipynb` as a documented dead-end. The crawler itself works, but its output isn't suitable for the tutoring application.

### Phase 2: Hand-curated ontology (Volume 1 only)

To validate the right node/edge ontology, the next phase was a fully manual graph of OpenStax Calculus Volume 1: ~100 concepts, ~250 edges, every relationship hand-typed with a justifying reason.

This phase was crucial for establishing what a "good" edge looks like — see [ontology.md](ontology.md). The output is preserved in `notebooks/02_calculus_kg_manual.ipynb`.

### Phase 3: LLM-assisted generation (full graph)

With the ontology established, the full graph (Volumes 1–3) was generated by prompting Claude to produce concept lists and typed edges section-by-section, given:

- The current section name and chapter context
- A rolling window of recently generated concepts (so cross-section prerequisite edges can reference them)
- A strict JSON output format and explicit conservativeness rules (most concept pairs should have no edge)

Cross-volume edges (e.g., "Chain Rule" → "Multivariable Chain Rule") were specified by hand and resolved against the generated concept names via fuzzy matching.

The resulting graph has **882 concepts** and **1,225 edges** across all three volumes.

### Quality control

After generation, the pipeline performs three cleanup steps:

1. **Deduplication** — concept names normalized; duplicate edges removed.
2. **Orphan edge removal** — edges whose endpoints don't exist in the concept set are dropped.
3. **DAG validation** — the subgraph of `prerequisite` edges is checked for cycles (which would be a logical error).

For the purposes of an honors thesis, the next quality-control step is **expert review**. A subset of edges should be sampled and validated by a math instructor; misclassifications should be tracked to estimate precision. This is planned work.

## Tutor architecture (RAG)

The tutoring application uses a two-step retrieval-augmented generation (RAG) pipeline:

```
Student question
      │
      ▼
[LLM call 1: Routing]
   "Given this question, which of these 880 concept names is the best match?"
      │
      ▼
Matched concept name
      │
      ▼
[Graph lookup]
   Pull concept description, hint, prerequisites, related concepts
      │
      ▼
[LLM call 2: Guidance]
   "Given this concept context, give Socratic guidance — never solve."
      │
      ▼
Response shown to student
```

### Why two LLM calls instead of one?

A single LLM call given the entire concept dictionary would (a) blow past context windows when scaling beyond calculus, and (b) tempt the LLM to ignore the graph and answer from its general training. Separating routing from guidance forces grounding: by the time the second call happens, the relevant graph context has already been retrieved, and the LLM's job is narrowed to *using* it.

### Why not fine-tuning?

A fair question: why not train a model on the graph rather than retrieve from it at runtime?

Two reasons. First, fine-tuning a useful LLM requires substantially more data than 882 nodes — typically tens of thousands of training examples. Second, RAG is more transparent: every response is grounded in specific, inspectable graph nodes that the student can navigate to. Fine-tuning would embed the graph implicitly in model weights, making it impossible to debug why the model said what it said. For an educational application, the audit trail matters.

## Limitations and known issues

- **LLM-generated concept names sometimes drift** across sections (e.g., "U-Substitution" vs. "U-Sub" vs. "Substitution Rule"). The deduplication step catches obvious duplicates by exact match, but semantic duplicates remain. A canonicalization pass using sentence embeddings would help.

- **Prerequisite direction isn't validated against ground truth.** The current graph encodes Claude's judgment about which concept is the prerequisite. A planned validation step compares this against textbook ordering (which concept appears first in OpenStax) and ultimately Korean curriculum ordering during winterim fieldwork.

- **"Difficulty" ratings are LLM-assigned and not calibrated.** They're useful for sorting nodes but shouldn't be treated as a rigorous measure without further work.

- **No edge weighting yet.** Currently every prerequisite edge is binary. A weighted version (e.g., "strong prerequisite" vs. "loose dependency") would be more pedagogically useful.

## Reproducibility

To regenerate the graph from scratch:

```bash
cd notebooks/
jupyter notebook 03_calculus_kg_full.ipynb
```

Set your Anthropic API key in cell 2 (`API_KEY = "..."`) and run all cells. The notebook saves progress after every section, so it's safe to interrupt and resume.

Approximate cost: $3–6 in API usage with Claude Sonnet 4.5.
Approximate runtime: 15–25 minutes.
